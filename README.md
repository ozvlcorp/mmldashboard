# MML — Аналитика для МойСклад

Виджет для МойСклад: складская аналитика + ABC / XYZ / RFM анализ. Реализован на Next.js 16, разворачивается одной командой на Vercel.

## Возможности

### 📦 Складская аналитика
Полная репликация формул из эталонной таблицы «ОМБОР аналитикаси» с исправлением замеченных багов исходника:
- Маржа, наценка, оборачиваемость
- Запас в днях vs норматив
- **Потери из-за OOS** (out of stock) — что недозаработали
- **Замороженные деньги** — что лежит на складе сверх норматива
- Бизнес-итог: возможная / фактическая / потерянная прибыль за горизонт N дней
- MML-флаг (Must Move List) с настраиваемым порогом

### 📊 ABC-анализ
Классификация позиций по вкладу в выручку/маржу. Группа A — ~80% метрики, B — следующие 15%, C — 5%. Парето-диаграмма с накопленной долей.

### 📈 XYZ-анализ
Стабильность спроса через коэффициент вариации (CV).
- X (CV ≤ 10%) — стабильный, можно работать «точно в срок»
- Y (CV 10–25%) — умеренно колеблется
- Z (CV > 25%) — нерегулярный, обязателен страховой запас

Совмещение ABC×XYZ даёт матрицу 9 групп — от AX (флагманы) до CZ (кандидаты на вывод из ассортимента).

### 👥 RFM-анализ
Сегментация клиентов по 3 метрикам:
- **R** (Recency) — давность последней покупки
- **F** (Frequency) — частота покупок
- **M** (Monetary) — суммарная выручка

9 сегментов: от Чемпионов (555) до Потерянных (111). Цвета и приоритет по выручке сегмента.


## Деплой на Netlify

В корне лежит `netlify.toml` с настроенным `@netlify/plugin-nextjs` — он автоматически разворачивает API-роуты как Netlify Functions, статику как CDN-кэш, и применяет заголовки для встраивания в iframe МойСклад.

### Шаги (CLI, рекомендуется)

```bash
# 1. Войти в Netlify (откроется браузер)
npm run netlify:login

# 2. Привязать проект — создаст новый сайт или подключит к существующему
npm run netlify:init

# 3. Preview деплой (получите URL вида oy-analytics-abc.netlify.app)
npm run netlify:preview

# 4. Когда всё хорошо — публикация в продакшен
npm run netlify:prod
```

### Через Git (CI/CD из репо)

1. Запушьте проект в GitHub / GitLab / Bitbucket.
2. На [app.netlify.com](https://app.netlify.com) → *Add new site → Import from Git*.
3. Выберите репозиторий. Netlify прочитает `netlify.toml` автоматически.
4. В *Site configuration → Environment variables* добавьте переменные (см. ниже).
5. *Deploys → Trigger deploy*.

### Переменные окружения

| Переменная | Назначение |
|------------|---------|
| `MOYSKLAD_APP_UID` | UID приложения из личного кабинета вендора МойСклад |
| `MOYSKLAD_SECRET_KEY` | Секрет для проверки JWT, выданного МойСклад виджету |
| `NEXT_PUBLIC_APP_URL` | Публичный URL приложения, например `https://oy-analytics.netlify.app` |
| `TELEGRAM_BOT_TOKEN` | (опц.) Токен бота для отправки напоминаний должникам |

Добавить через CLI: `netlify env:set MOYSKLAD_APP_UID your-uid`. Или из дашборда.

### Что делает Netlify-плагин

- API-роуты (`/api/...`) → Netlify Functions (serverless)
- `/widget` (SSR с проверкой JWT) → Edge / Function
- Главная `/` (static) → CDN
- `app/icon.svg` → favicon
- Заголовки `X-Frame-Options: ALLOWALL` + CSP — чтоб виджет открывался в iframe МойСклад

## Интеграция с МойСклад

### Вариант 1 — Vendor App (production)
1. Зарегистрируйте приложение в [личном кабинете вендора МойСклад](https://online.moysklad.ru/vendor/).
2. В дескрипторе укажите:
   - `Vendor API base URL` → `https://your-app.vercel.app/api/moysklad`
   - `Widget URL` → `https://your-app.vercel.app/widget`
3. Скопируйте сгенерированный `appUid` и `secretKey` в переменные окружения Vercel.
4. Установите приложение на свой аккаунт МойСклад через каталог.

При открытии виджета МойСклад добавит в query JWT-параметр `contextKey`, который сервер проверяет через `MOYSKLAD_SECRET_KEY`. См. `lib/moysklad/widget.ts`.

Шаблон дескриптора: `public/moysklad-descriptor.json`.

### Вариант 2 — Прямой Bearer токен (для пилота)
Сделайте POST на `/api/moysklad/analytics`:

```bash
curl -X POST https://your-app.vercel.app/api/moysklad/analytics \
  -H "Content-Type: application/json" \
  -d '{
    "token": "ВАШ_BEARER_ТОКЕН_МОЙСКЛАД",
    "periodDays": 30,
    "normDays": 10,
    "normDaysAttribute": "Норматив запаса (дни)"
  }'
```

Получите готовые входы для всех 4 анализов. Токен можно сгенерировать в МойСклад: *Профиль пользователя → Сервис → Получить токен доступа*.

### Норматив запаса как дополнительное поле товара

В МойСклад нет встроенного поля «норматив запаса в днях». Чтобы виджет считал индивидуальный норматив на каждый SKU, создайте дополнительное поле:

1. *Товары → Настройки → Дополнительные поля → Добавить* для сущности «Товар».
2. Тип поля — **Число (целое)** или **Число (дробное)**.
3. Назовите его, например, `Норматив запаса (дни)`.
4. Заполните значение для нужных товаров.
5. В запросе к виджету передайте имя поля (или его UUID) в параметре `normDaysAttribute`.

Для товаров без заполненного поля будет использовано значение из `normDays` (фолбэк, по умолчанию 10).

Получить список существующих атрибутов можно так:

```bash
curl https://your-app.vercel.app/api/moysklad/attributes \
  -H "Authorization: Bearer ВАШ_ТОКЕН"
```

Эндпоинт вернёт массив с `id`, `name`, `type` и флагом `suitableForNorm` (true для числовых полей — их можно использовать как норматив).

## Архитектура

```
app/
├── page.tsx                # Главная (demo)
├── widget/page.tsx         # Точка входа из МойСклад iframe
└── api/moysklad/
    ├── analytics/route.ts  # Загружает данные и считает метрики
    └── vendor/route.ts     # Колбэки активации/деактивации

lib/
├── analytics/
│   ├── inventory.ts        # Формулы складской аналитики
│   ├── abc.ts              # ABC классификация
│   ├── xyz.ts              # XYZ + ABC×XYZ матрица
│   └── rfm.ts              # RFM сегментация
└── moysklad/
    ├── client.ts           # JSON API 1.2 клиент
    ├── transform.ts        # МойСклад → форматы аналитики
    ├── widget.ts           # JWT валидация для iframe
    └── types.ts

components/
├── dashboard.tsx           # Главные табы
├── views/                  # Отдельная view на каждый анализ
├── data-table.tsx          # Сортируемая таблица
└── ui/                     # shadcn-style примитивы
```

## Что было исправлено vs исходная таблица

1. **E20 (рост прибыли при OOS=0)** — было `E17/E19` (множитель 2.28×). Теперь рядом считается и процент роста `+127.75%`, и кратность.
2. **SUM(L6:L14)** и **SUM(O6:O14)** — диапазоны начинались со строки 6, теряя строку 5. Исправлено на `L5:L14` / `O5:O14`.
3. **Q (% замороженных)** — был только в итоге; теперь считается построчно.
4. **MML флаг** — раньше выставлялся вручную. Теперь автоматический по порогу доли в валовом доходе (по умолчанию 15%).
